# 轻量级模型库 — 渐进式实施计划

## 一、核心目标

用 Python (FastAPI) 构建一个独立的轻量级模型库服务，核心能力：

| 功能 | 说明 |
|------|------|
| 模型注册 | 登记模型基本信息（名称、框架、架构等） |
| 部署绑定 | 绑定已有 Docker 容器 / 绑定镜像 / 两者同时 |
| 容器生命周期 | start / stop / restart / rebuild |
| 推理代理 | 提供完整推理地址，调用方无需关心容器细节 |
| 健康检查 | 启动后轮询确认就绪 + 运行中心跳检测 |
| 日志查看 | 容器日志，支持 SSE 流式推送 |
| 状态同步 | 定时校验数据库与 Docker 实际状态一致性 |
| 资源监控 | CPU / 内存 / GPU 使用率 |

---

## 二、技术栈

| 组件 | 选型 | 说明 |
|------|------|------|
| Web 框架 | FastAPI | 异步、自动生成 OpenAPI 文档 |
| ORM | SQLAlchemy 2.0 + Alembic | 数据库操作和迁移 |
| 数据库 | MySQL | 业务数据存储 |
| Docker 客户端 | docker-py | 本地 Unix socket 直连 Docker daemon |
| 定时任务 | APScheduler | 状态同步、健康检查心跳 |
| 日志流 | sse-starlette | 容器日志流式推送 |

## 三、数据库设计

### 3.1 表关系

```
model_registry (模型注册表)
  │
  ├── 1:1 ── model_deploy_binding (部署绑定表)
  │
  ├── 1:1 ── model_io_schema (IO Schema，可选)
  │
  └── 1:N ── model_operation_log (操作日志)
```

### 3.2 模型状态机

```
stopped (默认) → starting → running → stopping → stopped
                  ↘         ↘ error   ↗
```

| 状态 | 含义 | 触发条件 |
|------|------|----------|
| `stopped` | 已停止 | 初始状态 / docker stop 成功 |
| `starting` | 启动中 | docker start/run 执行开始 |
| `running` | 运行中 | 容器 Running + 健康检查通过 |
| `stopping` | 停止中 | docker stop 执行开始 |
| `error` | 异常 | 启动失败 / 健康检查超时 / 容器崩溃 |

### 3.3 绑定类型

| bind_type | container_id | image_name | 场景 | 首次启动行为 |
|-----------|-------------|------------|------|-------------|
| `container` | 必填 | 可选 | 已有容器，直接纳入管理 | docker start |
| `image` | 空 | 必填 | 仅有镜像，启动时创建容器 | docker run → 自动填入 container_id，bind_type 改为 both |
| `both` | 必填 | 必填 | 容器已存在 + 记录镜像来源 | docker start（可 rebuild 用镜像重建） |

### 3.4 推理地址拼接

```
推理地址 = http://{host_ip}:{host_port}{inference_path}
```

示例：
- `http://127.0.0.1:8080/predict`
- `http://127.0.0.1:11434/v1/chat/completions`

**container_config 结构：**

```json
  "privileged": false,         // 特权模式
  "ulimits": [],               // 资源限制
  "labels": {},                // 容器标签
  "restart_policy": {}         // 重启策略
}
```

**优势：**
- 新增配置不需要改表结构
- 支持任意 Docker 参数
- `extra = "allow"` 允许自定义字段

**验证标准：**

### 阶段 6.5：模型推理接口（已完成）

**目标：** 提供直接运行模型的接口，无需手动管理容器生命周期

#### 接口设计

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/model-registry/{id}/infer` | 推理（容器必须已运行） |
| POST | `/api/model-registry/{id}/run` | 一次性运行（自动启动→推理→停止） |

#### 1. 推理接口 `/infer`

**逻辑：**
- 检查模型状态必须为 `running`
- 检查容器实际状态必须为 `running`
- 不满足条件直接报错，不自动启动
- 转发请求到模型的推理地址
- 返回推理结果

**请求体：**
```json
{
  "request_data": {
    "model": "qwen",
    "messages": [...],
    "temperature": 0.15,
    "max_tokens": 4096
  }
}
```

**响应：** 透传模型服务的响应

**错误场景：**
- 模型未绑定 → 404 "模型未绑定部署配置"
- 模型未运行 → 400 "模型未运行，请先启动"
- 容器异常 → 500 "容器状态异常"
- 推理服务不可达 → 502 "推理服务不可达"

#### 2. 一次性运行接口 `/run`

**逻辑：**
- 自动启动模型（如果未运行）
- **探活：轮询健康检查接口等待服务就绪**
- 转发推理请求
- 自动停止模型
- 返回推理结果

**请求体：**
```json
{
  "request_data": {
    "model": "qwen",
    "messages": [...],
    "temperature": 0.15,
    "max_tokens": 4096
  }
}
```

**查询参数：**
- `wait_timeout`: 等待服务就绪的超时时间（秒，默认 120）

**响应：**
```json
{
  "code": 200,
  "data": {
    "inference_result": { ... },
    "runtime_info": {
      "auto_started": true,
      "start_time": "2026-07-13T10:00:00",
      "stop_time": "2026-07-13T10:00:05",
      "duration_ms": 5000
    }
  }
}
```

**错误场景：**
- 模型未绑定 → 404
- 启动失败 → 500 "自动启动失败"
- 健康检查超时 → 504 "模型启动超时"
- 推理失败 → 502

#### 探活机制

1. 启动容器后，轮询 `health_check_url`
2. 每 2 秒检查一次
3. 超时时间可配置（默认 120 秒）
4. 成功条件：HTTP 状态码 < 400
5. 超时后自动停止容器并返回错误

## 七、注意事项

### 7.1 Docker SDK 相关

- `docker.from_env()` 默认连接 `unix:///var/run/docker.sock`
- 进程用户需要 docker 组权限：`sudo usermod -aG docker $USER`
- `container.stats(stream=False)` 首次调用可能返回空数据
- `container.logs()` 返回 `bytes`，需要 `.decode("utf-8")`
- GPU 支持需要 NVIDIA Container Toolkit

### 7.2 数据库相关

- `runtime_status` 的状态变更需要原子操作
- `model_deploy_binding` 的 `model_id` 是 UNIQUE 的
- 级联删除：删除 model_registry 时自动删除关联数据

### 7.3 定时任务相关

- APScheduler 在 FastAPI lifespan 中管理启停
- 状态同步需要避免与接口操作并发冲突
- 建议加锁或用乐观锁（检查 update_time）

### 7.4 SSE 流式日志

- 需要安装 `sse-starlette`
- 客户端断开时需要正确关闭 Docker 日志流
- 日志可能包含非 UTF-8 字节，需要 `errors="replace"`
